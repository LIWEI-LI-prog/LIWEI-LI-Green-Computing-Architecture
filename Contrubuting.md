# Contributing to LIWEI LI Green Computing Architecture

Thank you for your interest in advancing sustainable computing! This project thrives through collaborative research and development. We welcome contributions from researchers, engineers, students, and enthusiasts worldwide.

## Our Philosophy

### Open Innovation
We believe fundamental computing breakthroughs should be accessible to all. By open-sourcing this architecture, we aim to:
- Accelerate research through collaboration
- Ensure broad accessibility and adoption
- Build a diverse ecosystem of implementations
- Drive rapid innovation through shared progress

### Contribution Values
- **Scientific Rigor**: Contributions should be based on sound engineering principles
- **Practical Impact**: We prioritize work that moves the architecture toward real-world implementation  
- **Respectful Collaboration**: All discussions should be technically substantive and respectful
- **Transparent Progress**: Share both successes and failures to help others learn

## Contribution Areas

### Priority Research Areas

#### High Priority
- **Ternary Logic Simulation**: Accurate models of ternary gates and circuits
- **TRISC-V Toolchain**: Compiler and simulator extensions
- **Photonic Component Models**: Optical modulator and waveguide simulations
- **Thermal Analysis**: 3D stack cooling simulations

#### Medium Priority  
- **Memory Architecture**: Ternary PCM models and controllers
- **AI Algorithm Adaptation**: Ternary neural network implementations
- **Benchmarking Suite**: Performance comparison frameworks
- **Educational Materials**: Tutorials and explainer content

#### Exploration Areas
- **Alternative Ternary Technologies**: Beyond CMOS approaches
- **Novel Applications**: Domain-specific optimizations
- **Ecosystem Integration**: Compatibility with existing tools

### Contribution Types

#### Technical Contributions
- **Architecture Specifications**: Detailed design documents
- **Simulation Models**: Verilog, SystemC, or Python implementations
- **Software Tools**: Compilers, simulators, analysis tools
- **Test Benches**: Validation and verification suites

#### Research Contributions
- **Performance Analysis**: Theoretical or simulated performance studies
- **Comparative Studies**: Against alternative architectures
- **Scalability Analysis**: Scaling laws and limitations
- **Implementation Studies**: Fabrication and integration approaches

#### Community Contributions
- **Documentation**: Tutorials, explanations, API documentation
- **Code Reviews**: Helping review others' contributions
- **Mentorship**: Guiding new contributors
- **Outreach**: Spreading awareness and growing the community

## Contribution Process

### 1. Pre-contribution Discussion
For significant contributions:
1. **Open an Issue**: Describe what you want to work on
2. **Get Feedback**: Discuss with maintainers and community
3. **Design Review**: For architectural changes, get design approval
4. **Implementation Plan**: Outline your approach and timeline

### 2. Development Guidelines

#### Code Standards
- **Language**: Use English for all code and comments
- **Style**: Follow language-specific style guides
- **Documentation**: Document public APIs and interfaces
- **Testing**: Include tests for new functionality

#### Research Standards
- **Reproducibility**: Provide enough detail to reproduce results
- **Transparency**: Share both positive and negative results
- **Citation**: Properly attribute prior work
- **Data**: Share data where possible and practical

### 3. Submission Process

#### Pull Request Requirements
1. **Clear Description**: What changes are made and why
2. **Reference Issues**: Link to related issues or discussions
3. **Testing Evidence**: Show that your changes work
4. **Documentation Updates**: Update relevant documentation
5. **Review Readiness**: Code is ready for review

#### Review Process
1. **Automated Checks**: CI/CD pipelines run automatically
2. **Maintainer Review**: At least one maintainer reviews
3. **Community Feedback**: Others in community can comment
4. **Iteration**: Address feedback through updates
5. **Merge**: Once approved, maintainer merges

### 4. Post-acceptance

#### Recognition
- **Contributor List**: Added to project contributors
- **Hall of Fame**: Significant contributors recognized
- **Paper Authorship**: Research contributions may lead to co-authorship
- **Speaking Opportunities**: Invitations to present at community events

## Communication Guidelines

### Where to Communicate

#### GitHub Issues
- Bug reports
- Feature requests
- Technical questions

#### GitHub Discussions
- Research discussions
- Design debates
- Community Q&A
- Project announcements

#### Real-time Chat
- Quick questions
- Informal discussions
- Coordination of work

### Communication Standards

#### Technical Discussions
- Be specific and provide context
- Use code snippets, diagrams, or equations when helpful
- Reference relevant literature or prior art
- Stay focused on technical merits

#### Code Reviews
- Be constructive and specific
- Explain the "why" behind suggestions
- Respect different approaches and styles
- Focus on learning and improvement

#### Disagreement Resolution
- Focus on technical facts and data
- Consider multiple perspectives
- Seek to understand before being understood
- Escalate to maintainers if needed
